## DEC-001: Monolith-first architecture

**Date**: 2026-07-09
**Decision**: Build Knottix as a single Next.js application. No microservices, no separate backend.
**Context**: Team is small (5 people). Operational complexity of distributed systems would slow development without measurable benefit at current scale.
**Rationale**: A monolith with clean module boundaries is faster to build, easier to debug, and simpler to deploy. If a module needs to scale independently later, extract it then â€” not before there's evidence.
**Status**: Active

## DEC-003: Multi-provider AI with Claude as primary

## DEC-011: Database-driven RBAC, zero hardcoded permissions

**Date**: 2026-07-09

Adding a role or permission is a database operation, not a code change.

## DEC-014: Auth.js v5 with JWT strategy and credentials provider

## DEC-017: Permission-based route middleware

**Date**: 2026-07-09
**Decision**: Middleware reads permissions from the JWT token and checks against a module-to-permission mapping. No database queries in middleware.
**Context**: Next.js middleware runs on Edge runtime. Prisma client (with pg adapter) requires Node.js runtime. Middleware must be fast and stateless.
**Rationale**: The `authorized` callback in Auth.js handles authentication (logged in or not). Custom middleware then checks route-level permissions from the JWT. This keeps middleware Edge-compatible and sub-millisecond. Scoped permission checks (row-level) happen in server actions and API routes where the full Prisma client is available.
**Status**: Active

## DEC-018: Split auth config for Edge/Node runtime isolation

**Date**: 2026-07-15
**Decision**: Auth configuration is split into two files: `auth.config.ts` (Edge-safe, no Prisma imports) used by middleware, and `config.ts` (full config with providers and JWT/session callbacks importing Prisma) used by API routes and server actions.
**Context**: The middleware runs on Edge runtime where `node:path` and `node:url` (required by Prisma's generated client) are unavailable. A single config file caused the entire Prisma dependency chain to be bundled into the Edge middleware bundle.
**Rationale**: Standard Auth.js v5 pattern for Next.js. `auth.config.ts` exports only the `authorized` callback, `pages`, `session` strategy, and `trustHost`. `config.ts` spreads from `auth.config.ts` and adds credentials provider and JWT/session callbacks that query the database. Middleware creates its own `NextAuth(authConfig)` instance from the edge-safe config. The API route and server actions use `fullAuthConfig` through `index.ts`.
**Status**: Active

## DEC-019: Structured error system with AppError class

**Date**: 2026-07-15
**Decision**: All server-side errors use `AppError` class with typed error codes (`UNAUTHORIZED`, `FORBIDDEN`, `NOT_FOUND`, `VALIDATION`, etc.) and HTTP status mapping. Server actions return `ServerActionResponse<T>` via `actionHandler()`.
**Context**: Coding rules specify server actions return `{ success, data?, error? }` and errors should not be thrown for business logic. Raw `Error` throws were used in session guards â€” inconsistent with the standard.
**Rationale**: `AppError` provides factory methods (`unauthorized()`, `forbidden()`, `notFound()`, etc.) for consistent error creation. `actionHandler()` wraps async operations, catches `AppError` vs unexpected errors, and returns the standardized response. Internal errors are logged; business errors are surfaced to the client.
**Status**: Active

**Date**: 2026-07-15
**Decision**: The Knottix Kernel uses factory functions (`createKernel`, `createContainer`, `createEventBus`, etc.) returning interface-typed objects. No classes, no inheritance. All subsystems composed via the `KernelInterface` facade.

The DI container uses symbol-based `ServiceToken<T>` for type-safe service resolution with circular dependency detection. Modules are loaded in topological dependency order. The lifecycle manager enforces a boot â†’ init â†’ ready â†’ shutdown sequence. Hooks support before/after interception with abort capability on any operation.

## DEC-025: Memory Engine â€” namespace-scoped knowledge with permission-gated access

**Date**: 2026-07-16
**Decision**: The Memory Engine (`src/lib/memory/`) defines how knowledge is represented, stored, organized, versioned, and accessed. It is NOT embeddings, vector search, or semantic search â€” those are future layers. Memory is organized into 11 namespaces (organization, workspace, department, team, project, client, document, knowledge, conversation, user, agent) each mapping to existing RBAC permissions. Every memory request resolves through Organization â†’ Workspace â†’ Permissions â†’ Role. Composite memory IDs use the format `namespace:scopeId:entryId`. Versioning uses optimistic concurrency with immutable revision history. Soft delete with restore support; hard delete restricted to founders only. Events emitted through the Kernel's event bus for cross-module integration.
**Context**: DEC-023 established the Kernel, DEC-024 established the AI Runtime. The Memory Engine is the third foundation layer. Future engines (Prompt, Tool, Agent) will read/write knowledge through it. The access layer maps memory namespaces to existing permission strings (e.g., `project` namespace requires `projects:read/write`, `client` requires `clients:read/write`) â€” no new permission types introduced.
**Rationale**: Follows the factory-function pattern (DEC-023). All subsystems (store, cache, index, versioning, permissions, events, resolver) are independently composable. The in-memory store is a reference implementation; the real store will be Prisma/PostgreSQL. The LRU cache (500 entries default) with scope-based invalidation avoids stale reads. The multi-key index enables fast lookups by tag, namespace, source type, scope, organization, workspace, and reference target without database queries.
**Status**: Active

## DEC-026: Intelligence Platform â€” five subsystems built as one cohesive layer

**Date**: 2026-07-16
**Decision**: The Intelligence Platform (`src/lib/prompt/`, `src/lib/tools/`, `src/lib/agents/`, `src/lib/workflows/`, `src/lib/intelligence/`) is built and composed together rather than as separate unrelated modules. ## DEC-027: Persistence layer â€” one pattern, applied per interface shape, not five bespoke solutions

**Date**: 2026-07-16
**Decision**: Every Sprint-2 registry gets Prisma-backed persistence, but the *mechanism* depends on the registry's existing interface, never by changing that interface:
1. **Memory Engine** â€” `MemoryStore`/`RevisionStore`/`SnapshotStore`/`StatsStore` were already async (Session 8). `createPrismaMemoryStore(db)` is a straight drop-in replacement for `createInMemoryStore()`.
2. **Prompt / Agent / Workflow registries** â€” these interfaces are synchronous by design (Session 9), and the engines built on top (`prompt-engine.ts`, `agent-engine.ts`, `workflow-engine.ts`) call them synchronously mid-flow. Each gets a **write-through decorator** (`createPersisted*Registry(base, db)`): the sync in-memory registry stays the hot-path runtime engine; mutating calls (`create`/`update`/`register`/`deactivate`) additionally fire-and-forget a Prisma write; a new `hydrate()` method (not part of the original interface) reloads persisted rows into the in-memory registry at boot via a new additive `seed()` method on each registry (bypasses `create()` so hydrate doesn't re-trigger persistence or assign new IDs/versions).
3. **Tool Registry** â€” same decorator shape, but tool *handlers* are JS closures and cannot be serialized to a database at all. Only metadata (name, description, category, parameters, permission, `isActive`) is persisted. `hydrate()` applies persisted `isActive` overrides to tools whose handlers are *already* registered in code at that boot â€” it never fabricates a callable tool from a DB row with no handler.
4. **Feature Flags** â€” same decorator pattern applied to the Kernel's existing (synchronous) `FeatureFlagManager`, persisting to the generic `Setting` model under a new `FEATURE` scope value.
5. **Provider Configuration** â€” no existing registry to decorate (`ProviderRegistryInterface` holds live provider instances and health state, not configuration data). A standalone `ProviderConfigRepository` reads/writes only non-secret operational settings (baseUrl, timeoutMs, maxRetries, rateLimitRpm) to the existing `Integration` model. API keys are never read from or written to the database â€” they remain environment-variable-only via `ProviderConfig.apiKeyEnvVar`, unchanged from DEC-024.
**Context**: The mission required "Repositories must replace in-memory implementations where appropriate while preserving existing interfaces." Sprint 2's registries were built synchronous-by-design; converting them to return `Promise`s would break every engine built on top of them (a real interface redesign, explicitly prohibited). The decorator pattern satisfies both constraints at once: genuine cross-restart persistence, zero breaking changes.
**Rationale**: One consistent architectural decision applied five times (with the two necessary exceptions â€” full-replace for already-async Memory, metadata-only for un-serializable Tool handlers â€” each with a stated reason) is better reuse than five different bespoke persistence strategies. `prisma/schema.prisma` was extended additively: new enums (`ToolCategory`, `MemoryEntryStatus`, `SettingScope.FEATURE`), new columns on `AIAgent`/`PromptTemplate`/`AIMemory` (all with defaults or nullable), and three new models (`PromptTemplateRevision`, `AIMemoryRevision`, `AIMemorySnapshot`, `AITool`) â€” no existing column was removed or renamed except `PromptTemplate.organizationId`, which became nullable to match the code's existing `organizationId: string | null` (global prompt templates), a strictly-widening change.
**Status**: Active

## DEC-028: Tool Resolution = auto-execute zero-argument tools into context, not a function-calling loop

**Date**: 2026-07-16

## DEC-037: Desktop Runtime â€” a Tool Engine provider, not a new engine; real OS/browser execution with two deliberate exclusions

**Date**: 2026-07-18
**Decision**: The Local Desktop Runtime (`src/lib/desktop-runtime/`) is a plain TypeScript module exposing `RegisterToolInput[]` arrays, registered into the existing Tool Engine via the same `registerTools()` helper every AI Employee's tools already use (`registerDesktopRuntimeTools()`, called from `ensureOrganizationReady()` alongside `registerAllAIEmployees`/`registerCommandCenterPrompt`) â€” no new registry, execution pipeline, or invocation mechanism. It has five parts:
1. **`exec.ts`** â€” `runCommand()`/`launchDetached()`/`launchDirect()`, all built on Node's `child_process.execFile`/`spawn` with argument *arrays*, never a shell-interpolated string â€” a commit message, search query, or contact name can never break out of its argument position and inject a second command, which is what makes the allowlisted git/dev tools (`git_commit`, `start_dev_server`, ...) safe without needing a general `run_shell_command` escape hatch (explicitly out of scope per the mission).
2. **`os-tools.ts`/`clipboard.ts`/`screenshot.ts`/`dev-tools.ts`** â€” real, executing Windows automation: app launching (`cmd /c start`, the standard Windows launch idiom), `taskkill` for `close_app`, PowerShell's `Get-Clipboard`/`Set-Clipboard` for clipboard access, .NET's `System.Drawing`/`System.Windows.Forms` for screen capture, a `user32.dll` P/Invoke (`SetForegroundWindow`/`ShowWindow`) for window focus, and fixed single-purpose git/npm commands for the developer-tool allowlist. No new npm dependency for any of this â€” PowerShell and .NET are already present on Windows.
3. **`browser/`** â€” real Playwright automation (`playwright` added as a new, explicitly-requested dependency) of an actual visible Chrome window (`headless: false`, `channel: 'chrome'`, so it drives the user's real installed Chrome binary) on a **dedicated, isolated persistent profile** (`~/Knottix/BrowserProfile`) â€” never the user's default Chrome profile with its existing banking/email/other logins. The user logs into WhatsApp Web/Instagram once (QR code scan) inside this Knottix-controlled profile; cookies persist across restarts from there.
4. **`browser/whatsapp.ts`** â€” `whatsapp_open` â†’ `whatsapp_find_contact` â†’ `whatsapp_type_message` (drafts only, never sends) â†’ `whatsapp_preview_message` (read-only) â†’ `whatsapp_send_message` (`metadata.requiresConfirmation: true`). The send step is the *only* mutating one; there is no bulk-send, no scheduled send, and no way to reach `whatsapp_send_message` without the Command Center's confirmation flow (DEC-035) surfacing the drafted text and recipient first.
5. **`voice/`** â€” real STT (Deepgram or OpenAI Whisper, whichever API key is configured, fetch-based, no SDK â€” consistent with every other provider adapter in this codebase) and real TTS (ElevenLabs if configured, otherwise Windows' built-in SAPI voice via PowerShell, which genuinely speaks through the machine's speakers). Wake-phrase detection is STT-based (transcribe short rolling audio chunks client-side, string-match "Hey Knottix" server-side) rather than a dedicated low-power acoustic wake-word engine like Porcupine â€” no wake-word model ships with this codebase (IDEA-039 tracks upgrading this if per-listen STT cost becomes a real concern).

Two capabilities the mission listed under "Desktop Tools" are **deliberately not implemented as raw OS-level primitives**: global mouse click/double-click/right-click/scroll and global "type into whatever has focus" keyboard input. Where the mission's examples actually need pointer/keyboard control (filling a WhatsApp message box, clicking an Instagram button, scrolling a feed), those actions are implemented instead as Playwright page-scoped calls (`page.click()`, `page.keyboard.type()`, `page.mouse.wheel()`) confined to the Knottix-controlled browser tab â€” never a global OS hook that could just as easily type a password into whatever window happens to have focus. `focus_window`/`switch_window` (bring a named app to the foreground) are the only "window control" primitives exposed at the OS level.
**Context**: This mission was a direct continuation of a prior turn where the user's first version of this request (unrestricted `run_shell_command`, autonomous WhatsApp sending with no confirmation, an always-executing voice loop) was declined for exactly these reasons. The user's revised mission added the git/dev tool allowlist and explicit confirmation requirements (sending messages, closing apps, deleting, git commit/push, restart/shutdown) addressing the shell-execution and autonomous-send concerns directly; it did not revisit the raw-keyboard/mouse point, so that boundary â€” stated to the user in the prior turn â€” was carried forward here rather than silently dropped or silently reinstated.
**Rationale**: Confining keyboard/mouse simulation to Playwright's page context is the one design choice that makes "browser automation" and "safe automation" the same thing here â€” a `page.click('button')` call can only ever affect the specific automated tab Playwright itself opened, so even a worst-case prompt-injection scenario (a malicious page's content somehow influencing a later tool call) is bounded to that one browser tab, never the user's actual desktop, password manager, or any other application. An isolated browser profile for WhatsApp/Instagram (rather than the user's real Chrome profile) is the same principle applied to session/credential isolation: Knottix automation never gains access to accounts the user didn't explicitly log into inside that profile.
**Status**: Active

## DEC-039: Goal Execution Engine â€” a planner + Workflow Engine consumer, not a sixth engine; dangerous steps always pause for confirmation, never auto-approved mid-plan

**Date**: 2026-07-19
**Decision**: The Goal Execution Engine (`src/lib/goal-engine/`) turns one free-text goal into a real, running Task Graph â€” but the "Task Graph" is not a new data structure: it IS a `WorkflowDefinition`, and executing/pausing/resuming it IS the Workflow Engine's own `execute()`/new `resume()`. Three additive extensions to the Workflow Foundation made this possible, all closing pre-existing gaps rather than building parallel infrastructure:
1. **Real `tool`/`agent`/`prompt` step handlers** (`src/lib/workflows/step-handlers.ts`, `registerCoreWorkflowStepHandlers()`) â€” closes IDEA-036, which had flagged since Session 17 that the Workflow Engine had zero handlers registered for these three step types. A 'tool' step calls `toolEngine.execute()`, an 'agent' step calls `agentEngine.execute()` against a real AI Employee, a 'prompt' step calls `promptEngine.render()` + `aiRuntime.complete()` â€” the exact same calls every other part of Knottix already makes, just reachable from a workflow graph now.
2. **Pause/resume for confirmation-gated steps** (`WorkflowExecutionStatus.WAITING_CONFIRMATION` â€” additive Prisma enum value; `WorkflowConfirmationRequired` signal a step handler throws; `WorkflowExecutor.resume(executionId, confirm)`) â€” when the 'tool' handler sees `tool.metadata.requiresConfirmation && !confirmed`, it throws instead of executing; the executor catches this specifically, stops the run, and returns `WAITING_CONFIRMATION` with the paused context held in a closure-scoped map keyed by execution ID. `resume()` either marks the step confirmed and continues from it, or (on decline) routes through the step's own `onFailure` edge â€” a declined step is treated as a real failure, never silently skipped.
3. **Per-step retry** (`step.config.retry.maxAttempts`, read directly in the executor's `runStep`) â€” the dumb, generic half of "retry until success or the retry limit is reached." The smart half (diagnose â†’ repair â†’ retry) is NOT built into the executor â€” it's expressed as an ordinary graph shape in `templates.ts` (an `onFailure` edge to a diagnostic 'prompt'/'agent' step, optionally looping back), keeping the Workflow Foundation itself free of AI-specific logic per the same discipline DEC-026 already established for it.

**Project Templates** (`src/lib/goal-engine/templates.ts`) are the actual planning intelligence â€” a fixed, hand-written, reviewable set of `WorkflowStepDefinition[]` builders (`deploy-project`, `sync-project`, `engineering-summary`, `project-status-report`, `website-plan`, `content-draft`), matched by a deterministic keyword heuristic first (`matchGoalTemplate()`), with an optional AI upgrade (`planner.ts`'s `planGoal()`, mirroring `command-center/classify.ts`'s exact discipline: heuristic always runs, AI only ever picks from the SAME fixed template list, never in Demo Mode, falls back to heuristic on any failure) â€” a goal can never execute a plan nobody wrote, because the AI never generates a workflow graph, only a template *key*.

**Reused, not duplicated**: AI Employee delegation is 'agent' steps calling the seven already-registered employees (no new AI, no "QA" employee invented â€” the mission listed QA among the specialized employees, but no such employee exists and building one is out of this session's scope; verification steps use existing employees + real tool re-checks instead, and this gap is stated honestly rather than faked â€” see the Not Yet Done note below). Memory reuse (`src/lib/goal-engine/memory.ts`) writes one real `memoryEngine.create()` entry per finished goal (tagged `goal-execution` + template key + outcome) and reads them back before planning to surface a track record in `GoalPlan.reasoning` â€” informational only, never changes which template runs. The Live Execution Panel (`/goals`, `GoalExecutionPanel.tsx`) polls `GET /api/goals/[executionId]` via TanStack Query â€” the first real consumer of `QueryProvider` (present since an earlier session, never wired to an actual query until now). The Command Center routes goal-shaped free text ("Build a website for ACCD") to `/goals?goal=...` via the EXISTING navigation intent (`router.ts`'s `detectGoalCommand()`) â€” no new Command Center execution path, no second router.
**Context**: The mission required a goalâ†’planâ†’executeâ†’verifyâ†’retry pipeline reusing the AI Runtime/Tool Engine/Workflow Engine/Prompt Engine/Memory/Agent Framework/Desktop Runtime/Command Center with no new engine, and explicitly asked how dangerous autonomous steps (git commit/push, deploy, delete, WhatsApp send) should behave mid-plan. Raised to the user directly before implementation (not decided unilaterally): pause-and-wait-for-confirmation-per-step (preserving DEC-037/038 with zero exceptions) vs. pre-approve-the-whole-plan-up-front vs. exclude dangerous tools from autonomous plans entirely. The user chose pause-and-wait â€” the same per-instance confirmation discipline every dangerous tool already had, now also enforced mid-goal, not just mid-command.
**Rationale**: Treating the Task Graph as a `WorkflowDefinition` (not a new type) and its execution as `workflowEngine.execute()`/`resume()` (not a new runner) is the direct consequence of "reuse the Workflow Engine" â€” closing IDEA-036 wasn't a nice-to-have side effect, it was the only way to build this honestly without a second execution pipeline. Two genuine gaps in Knottix's real tool surface were named honestly rather than papered over: (1) no tool exists to stage git changes, so `git_add` (non-destructive, same allowlist tier as `git_status`/`git_pull`, no confirmation needed) was added to `dev-tools.ts` â€” the one new tool this session added; (2) no tool exists for code scaffolding, image/logo generation, invoice generation, or presentation generation â€” goals naming these capabilities are classified `unsupported` at planning time and NEVER given a workflow to execute (`GoalPlan.unsupportedReason`), matching the mission's own "No fake execution" instruction to the letter rather than fabricating a "success" for a capability that doesn't exist.
**Status**: Active

## DEC-041: Task Sessions â€” a persisted wrapper around Goal Execution, real crash recovery via Workflow Engine rehydration, honestly unavailable in Demo Mode

**Date**: 2026-07-19
**Decision**: `src/lib/task-sessions/` is the one new module this session's mission scoped ("Knottix must now become capable of long-running autonomous work sessions"). A `TaskSession` is deliberately NOT a new execution engine or planner â€” it is a persisted database row (`TaskSession` Prisma model, mirroring `Workflow`/`WorkflowExecution`'s own shape) that wraps one or more rounds of the EXISTING Goal Execution Engine (`startGoal`/`resumeGoal`, Session 19/20), tracking which `WorkflowExecution`s it has spawned toward one objective, real artifacts those executions produced, and enough identity to survive a server restart.
1. **Real crash recovery, built at the Workflow Engine layer, not duplicated at the session layer.** Two Workflow Foundation gaps made "if Knottix crashes... the session must continue from the last successful step" false before this session: (a) `WorkflowExecutor` only ever persisted a `WorkflowExecution` row on completion/failure â€” a *paused* (`WAITING_CONFIRMATION`) execution existed only in an in-memory `Map`, invisible to the database entirely; (b) `resume()` had no path to reconstruct that in-memory state if it was lost. Both are fixed generically, benefiting every future `WorkflowDefinition` consumer, not just Task Sessions: `runFrom()` now calls `historyStore.record()` on pause too (fixed a latent duplicate-row bug in the in-memory `WorkflowHistoryStore.record()` while touching it â€” it now upserts by `executionId` instead of always pushing); `WorkflowHistoryStore` gained an optional `getByIdRemote()` (a real, direct database lookup, implemented only by the Prisma-backed decorator); `WorkflowStateManager` gained `seed()`; `WorkflowExecutor.resume()` now takes a `context: WorkflowContext` parameter and, when the in-memory paused closure is gone, reconstructs the workflow/step graph/execution state from persisted data and rebuilds a fresh context from it. `goal-engine/engine.ts`'s `resumeGoal()`/`getGoalStatus()` both gained the same graceful DB-fallback rather than throwing "not found" for a session that genuinely exists. This directly closes IDEA-042's core claim ("a goal paused mid-confirmation is lost on server restart") for the confirmation-pause case, which is the only instant a workflow is ever not actively executing code â€” there is nothing to lose.
2. **Session-level orchestration is a thin wrapper, not new logic.** `task-sessions/manager.ts`'s `startTaskSession`/`resumeTaskSession`/`continueTaskSession`/`cancelTaskSession`/`getTaskSessionProgress` all delegate the actual planning/execution/verification/retry to `startGoal`/`resumeGoal`/`getGoalStatus` â€” the session layer's only real job is: create/update the `TaskSession` row, map `GoalExecutionSummary.status` to the mission's own vocabulary (RUNNING/PAUSED/BLOCKED/COMPLETED/FAILED/CANCELLED â€” `UNSUPPORTED` maps to `BLOCKED`), and merge real artifacts. "Continue my website" (an objective that already finished, failed, or was blocked) calls `startGoal()` again with the SAME stored objective text for another round â€” the one place a session spans more than one Goal Execution â€” still zero new planning intelligence, and deliberately NOT an automatic "does this objective need more work" AI judgment (that would be the "another planner" the mission explicitly forbade); continuing across rounds is always a real user/Command Center action.
3. **Artifacts are extracted, never fabricated.** `GoalStepSummary` gained an `artifact: GoalStepArtifactHint | null` field, populated in `goal-engine/engine.ts` only when a step's real output object has a `.path`/`.url` string field, or its raw text contains git's own commit-hash line (`[branch abc1234] message`) â€” the exact format `git commit` prints on success. A step with no such field produces no artifact. `task-sessions/manager.ts` merges these (deduplicated by type+value) into `TaskSession.artifacts` after every round.
4. **Self-evaluation reuses the Memory Engine exactly like `goal-engine/memory.ts` already does** â€” `task-sessions/memory.ts`'s `recordSessionOutcome()` writes one real entry per terminal session (objective, outcome, round count, skills used, retries, artifact count, duration, failing steps), tagged `task-session` + skill keys + outcome; `recallSessionPatterns()` reads them back, surfaced on `/task-sessions` as an honest "N succeeded, M failed, averaging Xs" banner â€” informational, the same discipline `recallGoalHistory()` already established, not a hidden ranking algorithm.
5. **Demo Mode honestly declines Task Sessions entirely**, at both the API layer (`assertTaskSessionsAvailable()`, a 422 with a clear message) and the Command Center's `task_session:*` step handling â€” rather than building a second, in-memory-only session simulation mirroring `goal-engine/engine.ts`'s own demo path. A `TaskSession` is *defined* by being a real, restart-surviving database row; faking that would either be dishonest (claiming recoverability that doesn't exist) or exactly the "duplicate execution logic" the mission's own constraints forbid.
6. **Command Center control phrases** ("Continue my website", "Resume ACCD", "Show active work", "Pause current session") are new `TOOL_KEYWORDS` entries in `router.ts` (`task_session:continue`/`task_session:list`/`task_session:cancel`) resolved server-side in `command-center/engine.ts`'s new `runTaskSessionStep()` â€” which calls the exact same `task-sessions/manager.ts` functions the `/task-sessions` page itself uses, not a parallel implementation. `task_session:cancel` was added to `CONFIRMATION_REQUIRED_TOOLS` (stopping active work warrants the same confirmation gate as `close_app`).
**Context**: The mission required Task Sessions to survive a crash, browser reload, or server restart; to never fabricate progress; and to reuse the AI Runtime/Goal Execution Engine/Workflow Engine/Skill Registry/Tool Engine/Desktop Runtime/Browser Runtime/Memory/Agent Framework with no new AI, runtime, workflow engine, tool engine, or planner.
**Rationale**: Fixing crash recovery at the Workflow Engine layer (where the actual pause/resume state lives) rather than bolting a parallel persistence mechanism onto Task Sessions is the direct consequence of "no duplicate execution logic" â€” it also means IDEA-042's fix benefits the Goal Execution Engine and any future `WorkflowDefinition` consumer, not just this feature. Declining Demo Mode support outright, rather than quietly degrading to fake persistence, follows the same honesty discipline as `goal-engine`'s own `unsupported` plan classification (DEC-039) â€” a capability gap stated plainly is better than a capability that looks like it works but doesn't.
**Status**: Active

## DEC-042: Context Engine â€” a real orchestration layer over existing sources, the first step of every Goal Execution, honestly unavailable in Demo Mode

**Date**: 2026-07-20
**Decision**: `src/lib/context-engine/` is the one new module this session's mission scoped ("Before planning or executing any goal, Knottix must automatically discover every relevant piece of information already available inside the system"). It is deliberately NOT a new storage system, index, or AI â€” "reuse the existing Memory and Knowledge architecture... orchestrate them, not replace them" is implemented literally: every one of the 13 mission-named context sources (Memory, Projects, Tasks, Knowledge Base, Meeting Notes, Uploaded Documents, GitHub, Previous Task Sessions, Workflow History, Skill History, Generated Artifacts, Browser State, Desktop State) is a thin `Collector` (`collectors.ts`) that calls an already-existing real query â€” `memoryEngine.query()`, plain Prisma `db.project`/`db.task`/`db.meeting`/`db.file` reads, `listRepositories()` (GitHub service), `getSkillStats()`/`getSkillIndex()` (Skill Registry, Session 20), `listTaskSessionsForOrganization()` (Task Sessions, Session 21), `workflowHistoryStore.listByOrganization()`, and the real Desktop Runtime's `clipboardRead()`/`getOpenPageHandles()`. No new AI call is involved anywhere in collection or ranking.
1. **Ranking is a deterministic, inspectable formula, not a model.** Every collected item gets a `relevanceScore` (keyword overlap between the query and the item's title/summary, title matches weighted double â€” the same discipline `SkillDiscovery.discover()` already established in Session 20), a `freshnessScore` (exponential recency decay, 14-day half-life, neutral 0.5 for sources with no natural timestamp), and a `trustScore` (a static, hand-written per-source weight â€” curated Memory/Knowledge/GitHub/Project data trusted more than ephemeral Browser/Desktop state). `totalScore = relevance*0.5 + freshness*0.3 + trust*0.2`. `rankAndSelect()` splits the merged, sorted list into `selected` (top 15, above a relevance floor) and `rejected` (everything else), and every rejected item keeps a real reason ("below relevance floor" or "bundle already full") â€” nothing is silently dropped.
2. **"Automatic Project Detection" ("Continue ACCD" â†’ repo, branch, tasks, files, meetings, artifacts, without asking again) needed no special-cased logic.** It falls out of the generic pipeline: the goal text is scored against every collector in parallel, and a project named "ACCD" naturally surfaces high-relevance items from the project/task/meeting/document/github/task-session/artifact collectors simultaneously. Building a separate "project detection" mechanism would have been the exact duplicate system the mission's own constraints forbade.
3. **Integration point: after skill selection, before execution â€” not strictly "before planning" as the mission's pipeline diagram shows.** `goal-engine/engine.ts`'s `startGoal()` calls `collectContext(goal, user)` once the plan is known to be real (not `unsupported`), then `enrichAgentSteps()` prepends the selected context as a real, sourced text block to every 'agent' step's `config.input` before the workflow is created â€” an AI Employee reasoning over a goal is now grounded in real projects/tasks/memory/GitHub/session history, not just the raw goal text. Collecting context for a goal that turns out unsupported would be pure waste; this ordering is a deliberate, stated deviation from the mission's literal diagram, not an oversight (see the corresponding idea entry). `GoalExecutionSummary` gained `contextUsed: ContextItem[]`, round-tripped through `WorkflowContext.initialVariables.__contextUsed` the same way `__skillKeys` already was (Session 20) â€” no new persistence mechanism.
4. **Working Context is genuinely evolving, not a one-time snapshot.** `TaskSession` gained a `workingContext: Json` field (additive schema change). `task-sessions/manager.ts` merges each round's `GoalExecutionSummary.contextUsed` into it (deduped by source+ref, capped at 20, re-sorted by score) â€” reusing what `startGoal`/`resumeGoal` already collected rather than calling `collectContext()` a second time. "New artifacts should automatically become part of the session context" holds structurally: the `artifact` collector reads straight from `TaskSession.artifacts`, so the next `continueTaskSession()` round's collection pass picks up whatever the previous round just produced, with zero session-specific artifact-tracking code.
5. **Command Center**: "What do you know about ACCD?" / "Why did you choose this context?" / "What information are you using?" are new `context:explain` `TOOL_KEYWORDS` entries (`router.ts`), resolved in `command-center/engine.ts`'s `runContextStep()` by calling `explainContext()` â€” which re-runs the exact same `collectContext()`+ranking pipeline and formats a readable answer, so the explanation always describes what would ACTUALLY be collected right now.
6. **Mission Control**: `/context` (Context Inspector) â€” an ad-hoc query box showing Context Sources (every collector, including ones that found nothing), Selected Context, and Rejected Context, each item with all four scores and its real reason. `/task-sessions` shows each session's live Working Context inline.
7. **Memory**: `context-engine/memory.ts`'s `recordContextUsage()` writes one real entry per finished goal (tagged `context-pattern` + every contributing source + outcome) â€” called from `startGoal()` right after `recordGoalOutcome()`, satisfying "store frequently useful context combinations" without a second memory-writing mechanism.
8. **Demo Mode declines the Context Engine entirely** (`assertContextEngineAvailable()`, mirroring `task-sessions/demo-guard.ts`'s identical DEC-041 precedent exactly) â€” every collector is a real database/service query, and faking discovery of "real organizational context" would be a direct violation of "never fabricate context... every context item must originate from a real source."
**Context**: The mission required a Context Engine that automatically collects, ranks, and selects real context from 13 named sources before every Goal Execution, with a Context Inspector in Mission Control and Command Center question-answering, while reusing the AI Runtime/Memory/Goal Engine/Skill Registry/Workflow Engine/Task Sessions/Desktop Runtime/Browser Runtime with no new AI or duplicate systems.
**Rationale**: A dozen thin, real collectors behind one ranking formula is the direct consequence of "orchestrate, don't replace" â€” there was never a design considered where context got its own storage layer, since every source it reads already has one. Two genuine gaps in Knottix's real tool surface were named honestly rather than glossed over: (1) there is no tool that enumerates desktop state (running applications, window list) â€” the `desktop` collector honestly reports only the system clipboard, the closest real signal available, rather than fabricating a broader "desktop state" the mission's example implied; (2) context doesn't yet influence WHICH skill the planner selects, only how richly an already-selected skill's agent steps are grounded â€” extending `SkillDiscovery` to weight relevance by collected context is a natural next step, deliberately deferred to keep this session's scope to what "orchestrate existing systems" actually required.
**Status**: Active

## DEC-043: MCP Client Layer â€” external MCP servers as first-class Skills, cross-tenant safety fixed at three separate call sites

**Date**: 2026-07-21
**Decision**: `src/lib/mcp/` is the one new module this session's mission scoped ("Knottix should be able to communicate with external systems through the Model Context Protocol"). It is a real client, not a stub: the official `@modelcontextprotocol/sdk` (v1.29.0, justified as a new dependency the same way Playwright was in Session 18 â€” genuine protocol complexity, not a simple REST wrapper) drives a real handshake, capability negotiation, and `listTools`/`listResources`/`listPrompts`/`callTool`/`readResource`/`getPrompt` calls over stdio/SSE/HTTP transports.
1. **Server Registry** â€” `MCPServer`/`MCPCallLog` Prisma models (additive), `db/queries/mcp-server.ts` (auth tokens encrypted at rest via the SAME `encryptSecret`/`decryptSecret` GitHub PATs already use, DEC-034), `mcp/client.ts` (real `connectMCPClient()`/`fetchServerCapabilities()`), `mcp/manager.ts` (a `globalThis`-anchored live-connection map â€” IDEA-043's lesson applied proactively again â€” `discoverAndConnectServers()`, `refreshMCPServers()`, `callMCPTool()`/`readMCPResource()`/`getMCPPrompt()`, every one wrapped in `loggedCall()` so "log every MCP request" holds by construction, not by convention). A connection failure is never an exception the caller must guard against â€” `connectMCPServer()` always persists a real `CONNECTED` or `ERROR` status; "never fake a connected server" is enforced the same way Task Sessions/Context Engine enforce their own honesty constraints (DEC-041/042).
2. **Tool Integration â€” real Tool Engine tools, not a parallel calling mechanism.** `mcp/skills.ts`'s `registerMCPToolsAndSkills()` registers every connected server's tool as a REAL `intelligence.tools.register()` entry (handler calls `callMCPTool()` â€” the identical `toolEngine.execute()` path every local tool already goes through) and, where the input schema is simple enough to extract from free text (0 or 1 required param â€” 2+ is honestly left un-composed rather than guessing which argument goes where), a real `Skill` wrapping it. `goal-engine/planner.ts`'s `discoverBest()` merges the static, database-free `getSkillIndex()` catalog with the calling organization's own `SkillEngine` before picking the top-scoring match â€” "Local Skills and MCP Skills without knowing the difference" is literally one merged, re-sorted list.
3. **Three chained cross-tenant multi-tenancy bugs, caught during this session's own implementation before any user ever saw them** (not reported â€” self-discovered, matching this codebase's own documentation discipline): (a) MCP-derived skills were originally going to be registered into BOTH the org-scoped `SkillEngine` and the shared, `globalThis`-anchored, database-free `getSkillIndex()` â€” since the latter is ONE process-wide instance shared by every organization, this would have leaked one org's connected-server/tool names into every other org's goal planning; fixed by registering into `skillEngine` only, reuniting the two catalogs per-request in `discoverBest()` instead. (b) The registered Tool Engine tool name was originally keyed on `sanitize(server.name)` â€” since the Tool Registry is a flat, non-org-scoped `Map`, two organizations both naming their server "figma" would collide, silently routing org B's calls through org A's connection; fixed by keying `mcpToolName()` on `server.id` (a UUID) instead. (c) The registered skill's `RegisterSkillInput` initially omitted `organizationId`, defaulting every MCP skill into the `SkillRegistry`'s shared "global" bucket despite the registry itself being org-scoped internally; fixed by setting `organizationId: server.organizationId`. **A fourth, related bug was found one layer up, during this same session's follow-through**: `discoverBest()`'s call to the org `SkillEngine`'s `discover()` initially passed no filter object at all â€” `SkillDiscovery.discover(query, filter?)`'s underlying `find()` only applies organization scoping `if (filter.organizationId !== undefined)`, so the three fixes above would have been silently bypassed at this specific call site, mixing every organization's MCP skills back together at planning time. Fixed by threading an explicit `{ organizationId }` through `discoverBest()`, `planGoalHeuristic()`, and `planGoal()`'s `activeSkills` candidate list for the AI-upgrade path too.
